Project Hyphen · Onboarding Brief

性能与可靠性度量（Measurement & Reliability）任务清单

为统计学（statistics）背景的协作者划定的独立子领域 · 本地优先的 macOS ⇄ Android 配对层

为什么是这块？ Project Hyphen 本质是系统/网络/安全工程，没有数据科学内核。但它有一个真实且长期缺人的痛点：把"性能好不好、可靠不可靠"从口头印象变成有方法论的、可复现的度量。这正好是统计背景能独立认领、并产出作品集的切口。

定位要诚实： 这更接近 data / quality engineering，不是 ML / 量化建模。对想做"可靠性、性能、数据工程、分析"方向的简历含金量高；对想做纯算法/ML 的，含金量中等。

统计技能 真正用到统计 易 几天可上手 中 需读代码 难 需真机/系统知识

Track A — 传输性能基准 throughput / latency 分布化

A

把传输性能变成可复现、带置信区间的基准

对应 HyphenTransfer · create_large_transfer_fixture.py
编号任务用到的统计难度
A1
定义基准方法论
明确指标（吞吐 MB/s、端到端延迟、首字节时间）、预热轮次、重复次数、丢弃异常值规则。写成一页 methodology 文档。
 实验设计 采样/重复/变量控制 易
A2
构建可复现的基准 harness
用 create_large_transfer_fixture.py 生成 1 MB / 100 MB / 1 GiB 档位，跑 loopback 与真机两条路径，自动记录每轮结果为结构化数据（CSV/JSON）。
 分层抽样 文件大小×网络条件 中
A3
分布与区间分析
不报均值，报 p50 / p95 / p99、标准差、置信区间；画延迟分布直方图与 CDF。识别长尾来源。
 分位数/CI bootstrap 置信区间 易
A4
性能回归门禁
设定"相对基线退化多少算回归"的统计阈值（而非拍脑袋的固定百分比），接入 scripts/check.sh 思路做 CI gate。
 假设检验 变化点/阈值 中

Track B — 可靠性矩阵 投递率 / 重连 / 故障注入

B

用抽样和成功率量化"它到底有多可靠"

对应 HyphenNotifications · trust/pairing · resume 路径
编号任务用到的统计难度
B1
通知镜像投递成功率实验
设计 10+ app 的通知矩阵，定义"成功投递"判据，重复触发，估计每类 app 的投递率 + Wilson 置信区间，而不是"试几次都行"。
 比例估计 Wilson 区间 难 需真机
B2
睡眠/唤醒重连可靠性
配对设备反复进入睡眠再唤醒，测量重连成功率与重连耗时分布，定位失败模式。
 生存/耗时分布 难 需真机
B3
故障注入 + resume 正确性
在传输中注入断网/丢包/TLS 中断，验证 resume 后文件完整（哈希校验），统计各故障下的恢复率。
 故障×恢复列联表 中

Track C — 诊断数据分析 structured log → 分布 / 异常 / dashboard

C

把已有的结构化日志变成可视化与异常信号

对应 diagnostics/StructuredLog.kt · StructuredLog.swift · RedactedDiagnosticsExporter.kt
编号任务用到的统计难度
C1
梳理可分析的指标字段
读现有 StructuredLog，列出连接、重连、传输、错误等字段，定义一份"可分析指标"清单（保持脱敏边界）。
 指标设计 中
C2
日志解析 + 分布/异常检测
用 Python(pandas) 解析脱敏导出，做连接时长、错误率的分布分析，对异常会话做简单离群检测。
 离群检测 z-score/IQR 易
C3
本地 dashboard
做一个纯本地、脱敏的可视化页面（趋势、分布、Top 错误），符合"诊断本地优先"约束。
 数据可视化 易

Track D — 协议模糊与一致性 test-vectors / 覆盖率

D

统计化地生成协议输入并量化覆盖

对应 protocol/test-vectors/ · validate_protocol_fixtures.py
编号任务用到的统计难度
D1
property-based / 统计化输入生成
基于现有 envelope / transfer-manifest schema，生成大量有效与边界输入，喂给校验器找一致性漏洞。
 随机化测试 采样策略 中
D2
覆盖率统计
量化 test-vectors 对 schema 字段/取值空间的覆盖，找出未被测到的组合，补 fixture。
 覆盖度量 易

📄 简历上可以怎么写

  • Performance benchmarking: 为开源 macOS/Android 传输层设计可复现基准，用分位数与 bootstrap 置信区间刻画吞吐/延迟，建立性能回归门禁。
  • Reliability analysis: 设计通知投递与重连的抽样实验，用比例估计 + Wilson 区间量化可靠性，定位失败模式。
  • Fault-injection testing: 故障注入下验证断点续传正确性，统计各故障的恢复率。
  • Diagnostics analytics: 解析结构化日志做分布与离群检测，构建本地可视化 dashboard。

⚠️ 必须守住的边界（项目硬约束）

  • loopback ≠ 真机证据。 A/B/C 的真机部分必须真机跑，不能用回环结果冒充。
  • 诊断本地优先、默认脱敏。 不引入遥测/上传，不碰 .env/密钥/私密日志。
  • 区分"自动化不变量"与"现场证据"。 报告里要标注哪些是测试得出、哪些是真机观测。
  • 结论分级：confirmed / unsupported / disputed / environment-only。

建议上手顺序 由易到难、先出成果

  1. 第 1 步 · C2 + C3：从已有脱敏日志做分析与 dashboard。纯 Python，不碰系统代码，几天就能出第一个可展示成果，建立信心。
  2. 第 2 步 · A1 → A2 → A3：定方法论、搭基准 harness、做分布分析。这是整条线的核心作品集。
  3. 第 3 步 · D1 + D2：协议模糊与覆盖率，进一步理解 wire 行为，难度可控。
  4. 第 4 步 · A4 + B3：性能回归门禁与故障注入，开始接触工程集成。
  5. 第 5 步 · B1 + B2：真机可靠性矩阵——最难、最有分量，等前面铺垫好再上。
